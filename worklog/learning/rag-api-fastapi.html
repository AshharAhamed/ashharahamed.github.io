Introducing Today's Project!

In this project, I implemented a Retrieval-Augmented Generation (RAG) based chat endpoint to understand how modern AI chat applications work. I wanted to learn the complete process, from storing documents as embeddings in a vector database to retrieving relevant information and generating accurate responses using an LLM.

I'm interested in AI implementation because I enjoy building intelligent applications and understanding how the underlying technologies work. This project gave me practical experience with embeddings, vector databases, semantic search, prompt engineering, and LLM integration - important building blocks for enterprise AI solutions.

Key tools and concepts

Key tools used: FastAPI, ChromaDB, Ollama, and Swagger.

Key concepts learned: RAG, embeddings, and metadata filtering.

Challenges and wins

This project took approximately 4 hours. The most challenging part was building an intuition for what embeddings actually represent.

Performing RAG Manually

Before writing any code, I created a manual Retrieval-Augmented Generation (RAG) demonstration to understand how the process works end-to-end.

RAG is an AI architecture that enables a Large Language Model (LLM) to answer questions using external knowledge instead of relying only on the information it was trained on. It consists of three main stages:

  • Retrieval - search a knowledge source (documents, PDFs, or a vector database) to find the most relevant information related to the user's question.
  • Augmentation - combine the retrieved information with the user's query to create a richer prompt that gives the LLM the necessary context.
  • Generation - the LLM uses the augmented prompt to generate a response that is accurate, relevant, and grounded in the retrieved information.

Understanding the three parts of RAG

I performed a manual demonstration of RAG by adding relevant personal information directly into the prompt instead of retrieving it from a knowledge base. The retrieval step was performed manually by selecting the relevant personal details myself, the augmentation step involved including those details in the prompt, and the generation step was handled by the LLM to produce the final response. This exercise helped illustrate the core RAG workflow before implementing it in code.

Comparing the two AI models

One key difference I noticed is that the Qwen2.5:0.5B model is a Large Language Model (LLM) designed for understanding prompts and generating natural language responses, allowing conversational interactions.

In contrast, an embedding model (such as embed-text) does not generate text or participate in conversations. Instead, it converts text into high-dimensional numerical vectors (embeddings) that capture the semantic meaning of the text. These embeddings are primarily used for:

  • Semantic search
  • Similarity matching
  • Document retrieval
  • Vector database indexing
  • Retrieval-Augmented Generation (RAG)

In a RAG system, the embedding model converts both documents and user queries into vectors. These vectors are compared to find the most semantically relevant documents, which are then passed to the LLM (Qwen2.5:0.5B) to generate the final response.

Building a Personal Knowledge Base

Next, I created personal profile documents, split (chunked) the content into smaller pieces, generated embeddings for each chunk, and stored those embeddings in ChromaDB.

An embedding is a high-dimensional numerical representation of the semantic meaning of data, such as text, images, or audio. Instead of storing the original content for searching, the embedding model converts each chunk into a vector that captures its meaning. By storing these vectors in ChromaDB, I can perform semantic similarity searches: when a user submits a query, it's also converted into an embedding, and ChromaDB retrieves the most relevant chunks based on vector similarity. These retrieved chunks are then provided to the LLM as context for generating an accurate response.

Creating the profile document

I included information about myself in a personal profile document, which serves as the knowledge source for the RAG system. When a user asks a question related to my background, the RAG pipeline retrieves the relevant sections of the profile and provides them as context to the LLM - so the AI can answer based on my personal profile rather than relying only on its pre-trained knowledge.

How semantic search finds relevant chunks

When I ask a question, the embedding model first converts the question into a vector, just as it did for the document chunks. ChromaDB then performs a similarity search by comparing the query vector with the vectors stored in the database, identifying the chunks whose vectors are closest to the query vector in a high-dimensional vector space. These closest chunks are considered the most semantically relevant to the question - they're retrieved and passed to the LLM as context, enabling it to generate an accurate, context-aware response.

Creating the RAG API with FastAPI

Next, I built an API exposing an /ask endpoint. This endpoint accepts a user's question, retrieves the most relevant information from the knowledge base using the RAG pipeline, and generates a grounded answer using an LLM. Unlike a standard LLM response, the answer is based on the retrieved context, making it more accurate, relevant, and less likely to hallucinate. I verified the implementation using Swagger UI by sending different questions and validating that responses were generated from the retrieved knowledge base.

How the /ask endpoint works

When a question is received, the API first converts it into an embedding and queries ChromaDB to find the most relevant document chunks based on semantic similarity. The retrieved chunks are then included as context in the prompt sent to the LLM, which generates a grounded response based on the knowledge stored in the database rather than relying solely on its pre-trained knowledge.

Testing with Swagger UI

I tested the API by asking: "What is my name?"

The AI answered with: "The person's full name is Ashhar."

The context used was the relevant chunk retrieved from my personal profile stored in ChromaDB, which contained information such as "My name is Ashhar. I'm currently learning about cloud computing, AI, and DevOps. I enjoy sci-fi films and surfing the internet." The RAG pipeline retrieved this profile information and provided it to the LLM, allowing it to generate a grounded answer based on the stored context rather than the model's own knowledge.

Extending to a Multi-User AI Directory

In this extension, I added multi-user support, since it allows adding multiple user profiles and giving more personalized context based on the user behind the API. Multi-tenancy means a single application serving multiple distinct users/clients.

Adding the POST /documents endpoint

I added a POST endpoint that permits a user to add their own profile. Metadata filtering allows filtering context based on the user.

Verifying multi-user filtering

I tested multi-user queries by first querying without a user filter, then adding "Jordan" as a user filter. The filter worked correctly - only the relevant context chunks retrieved for that particular user came back, since chunks were tagged with user metadata when stored in ChromaDB.
